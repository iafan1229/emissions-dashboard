# decision.md — 설계 의사결정 기록

발표 시 "왜 이렇게 설계했는가"를 설명하기 위한 의사결정 기록.

---

## 0. [핵심] PCF 계산 불가 — 전과정 배출량 시각화로 해석

**결정**: PCF(제품 1개당 배출량)를 계산하지 않고, **PCF를 구성하는 전과정 활동 데이터의 배출량을 시각화**한다.

**이유**:
- PCF = 총 배출량 ÷ 생산 수량 인데, 과제 데이터에 생산 수량이 없음
- 월별 데이터를 보면 1월 722 kgCO₂e, 2월 1,571 kgCO₂e로 변동이 커서 1대 기준으로 보기 어려움
- 임의로 생산 수량을 가정하는 것은 근거 없는 수치를 만드는 것

**해석**:
> "PCF 전과정 데이터를 시각화" = PCF를 계산하는 것이 아니라,
> PCF를 구성하는 전과정(원료조달/제조/운송) 단계별 배출량을 시각화하여
> 어느 단계/Scope에서 배출이 집중되는지 파악할 수 있도록 한다.

**Trade-off**:
- 과제 제목과 구현 사이에 괴리가 생길 수 있음
- 발표에서 이 ambiguity를 발견하고 합리적으로 해석했다는 것을 명확히 설명할 것
- docx 평가 기준 "ambiguity를 발견하고 assumption을 문서화하라"에 부합

---

## 1. PCF 중심 대시보드로 방향 설정

**결정**: 기업 단위 GHG 대시보드가 아닌 **제품 단위 PCF 대시보드**로 구현한다.

**이유**:
- 과제 요구사항이 "PCF 전과정 데이터를 시각화"라고 명시
- 과제 데이터 자체가 CT-045라는 단일 제품의 생산 활동 데이터

**Trade-off**:
- 제품이 CT-045 하나뿐이라 제품 간 비교가 불가능
- 대신 전과정(LCA) 단계별 분석에 집중하여 PCF 개념을 충실히 반영

---

## 2. GHG Scope를 LCA 단계와 함께 표시

**결정**: PCF 전과정 시각화에서 LCA 단계별로 보여주되, **각 단계에 해당하는 GHG Scope를 함께 표시**한다.

**이유**:
- 평가 기준에 "PCF, GHG Scope 개념을 코드·설명에 반영했는가" 명시 (25%)
- PCF와 GHG Scope는 엄밀히 다른 프레임워크지만, 이 과제 데이터에서는 자연스럽게 매핑 가능

| LCA 단계 | GHG Scope |
|---|---|
| 원료조달 | Scope 3 |
| 제조 | Scope 2 |
| 운송유통 | Scope 3 |

**Trade-off**:
- 두 개념을 억지로 연결하면 도메인 혼란을 줄 수 있음
- 발표에서 "PCF는 LCA 기반, GHG Scope는 기업 기반으로 다른 프레임워크이나, 이 과제에서는 활동 데이터가 동일하므로 매핑하여 표시했다"고 명확히 설명할 것

---

## 3. 회사를 별도 엔티티로 분리

**결정**: Company를 Product에 종속시키지 않고 **별도 엔티티**로 분리한다.

**이유**:
- docx 과제의 데이터 모델에 Company 타입이 명시되어 있음
- 향후 여러 회사의 제품을 비교하는 확장성을 고려
- 회사별 배출량 비교 뷰를 대시보드에서 제공 가능

**Trade-off**:
- 현재 데이터는 CT-045(Acme Corp) 하나뿐이라 회사별 비교가 사실상 불가능
- seed data에 Globex(c2)를 추가하여 비교 기능을 시연 가능하도록 구성

---

## 4. 활동 데이터 목록 페이지 제거

**결정**: `/activities` 목록 페이지를 별도로 만들지 않는다.

**이유**:
- 활동 데이터는 이미 제품 상세 페이지의 차트로 시각화됨
- 별도 목록 페이지는 중복이며 과제 체크리스트에도 요구사항 없음
- 구현 시간 대비 평가 점수 효율이 낮음

**Trade-off**:
- 원시 데이터를 테이블로 확인하는 기능이 없음
- 필요 시 제품 상세 페이지 하단에 간단한 테이블로 추가 가능

---

## 5. Post를 리포트로 간주하고 별도 목록 페이지 제공

**결정**: Post를 **월별 지속가능성 리포트**로 간주하고, 작성은 제품 상세 페이지에서, 전체 목록은 `/reports` 페이지에서 확인할 수 있도록 분리한다.

**이유**:
- Post의 seed data가 `title: "Sustainability Report"`, `content: "Quarterly CO2 update"`로 리포트 성격이 명확함
- 경영진이 특정 월의 배출량을 보고 코멘트/리포트를 작성하는 용도
- 제품 상세 안에만 숨기면 전체 리포트를 한눈에 볼 수 없음
- 작성(제품 상세) ↔ 조회(리포트 목록) 분리가 UX상 자연스러움

**Trade-off**:
- 페이지가 하나 더 늘어남
- 제품 상세와 리포트 목록 간 상태 동기화 필요 (`createOrUpdatePost` 후 목록 갱신)

---

## 6. 총 배출량과 PCF를 구분하여 표시

**결정**: 대시보드에서 **"기간 총 배출량(kgCO₂e)"** 으로 표시하고, PCF는 계산하지 않는다.

**이유**:
- PCF는 제품 1개당 탄소 배출량 (단위: kgCO₂e / 개)
- 생산 수량 데이터가 없어 계산 불가 → #0 핵심 의사결정 참고
- "총 배출량을 PCF 근사값으로 표시"는 단위가 맞지 않는 도메인 오류

**Trade-off**:
- PCF 수치를 보여주지 못하지만, 잘못된 수치를 보여주는 것보다 낫다
- README Assumption에 명확히 명시

---

## 7. 배출계수를 별도 모듈로 분리

**결정**: 배출계수를 컴포넌트에 하드코딩하지 않고 **`lib/constants/emissionFactors.ts`로 분리**한다.

**이유**:
- 과제 데이터에 "배출계수는 DB에 별도 테이블로 관리하고 버전 이력을 추적하도록 설계하라"고 명시
- 배출계수는 정책/기준에 따라 변경될 수 있으므로 중앙 관리 필요
- 확장성: PostgreSQL 사용 시 DB 테이블로 마이그레이션 용이

**Trade-off**:
- 현재는 상수로 관리하므로 실제 버전 이력 추적은 미구현
- `version`, `effectiveDate` 필드를 타입에 포함하여 향후 확장 가능성을 열어둠

---

## 8. 사용/폐기 단계를 "데이터 없음"으로 명시

**결정**: LCA 전과정 5단계 중 사용/폐기 단계는 구현하지 않고 **"데이터 없음"으로 UI에 명시**한다.

**이유**:
- 과제 데이터에 사용/폐기 단계 데이터가 없음
- 없는 데이터를 없다고 표시하는 것이 도메인을 정확히 이해했다는 증거
- 전과정 5단계 프레임워크를 유지하면서 데이터 한계를 투명하게 표현

**Trade-off**:
- 완전한 PCF 전과정 분석이 아님
- README Assumption에 명시

---

## 9. Scope 1을 0으로 명시

**결정**: Scope 1(직접 배출) 데이터가 없더라도 **0 kgCO₂e로 명시**한다.

**이유**:
- Scope 1은 GHG 프레임워크의 기본 구성 요소로, 데이터가 없어도 항목 자체는 존재해야 함
- 실제 기업 탄소 보고서에서도 Scope 1이 0인 경우 명시적으로 표기
- "Scope 1이 없으니 표시 안 함"은 도메인 이해 부족으로 보일 수 있음

---

## 10. seed data를 example로 간주하여 확장 가능한 구조로 설계

**결정**: docx의 seed data는 고정된 데이터가 아닌 **초기 예시 데이터**로 간주하고, 활동 데이터 입력 폼을 통해 데이터를 추가할 수 있는 구조로 설계한다.

**이유**:
- docx에 "seed data example"이라고 명시되어 있음
- 회사가 고정된 대시보드보다 데이터를 추가/관리할 수 있는 구조가 확장성 평가에 유리
- 평가 기준 "확장성·재사용성·안정성을 고려한 설계" (30%) 반영
